# 可复用工程约定包（从 MEIS 沉淀）

> **用途**：把 MEIS 实践中已验证、可迁移到其他系统的约定集中在此，减少重复劳动。  
> **来源**：`docs/meis-requirements.md` 附录 Q / C / D / E / F / G / H / I / R / S / T 等。  
> **用法**：新系统可整份复制后，按「落地映射」改路径与模块名；MEIS 专属细节见文末附录。

**版本**：1.6（2026-07-15）

---

## 维护约定（强制）

本文件与 `docs/meis-requirements.md` **双写**：

| 变更发生在 | 必须同步 |
|------------|----------|
| `meis-requirements.md` 新增/修订**可跨项目**约定 | 同步抽象进**本文**（去掉 MEIS 路径后写通用原则） |
| 本文修订可复用约定 | 若 MEIS 仍适用，回写需求文档对应附录，并更新 Q.6 索引 |

**可复用 vs MEIS 专用**：

- **写入本文**：协作流程、问题分流、库表双轨、审计七列/软删、实体变更记录模式、字典/外键展示、草稿/提交/撤回、验收清单、Agent 规则。
- **不必全文复制**：具体业务字段表（如 T.5）、某一工单状态机细项、某次审计补全记录 → 仅在文末「落地映射」链回需求文档。

## 0. 文档与问题分流（建议固定）

| 类型 | 放哪里 | 含义 |
|------|--------|------|
| 功能/业务需求 | 需求正文或附录 | 要做的能力 |
| 缺陷 / 技术债 | 「已知问题」章 | 已发现、要修 |
| 产品未决 | 「待确认问题」章 | 需业务方拍板 |
| 已认可但暂缓 | 「待开发功能池」章 | 值得做，暂无时间开发/测试 |
| 协作与技术约定 | 本约定包 / 项目附录 Q | 怎么协作、怎么写代码 |

**编号建议**：

- 需求：`{模块}-{类型}-{序号}`（F 功能 / B 规则 / X 问题…）
- 暂缓：`BACKLOG-{模块}-{序号}`

## 1. 需求协作与交付（强制）

### 1.1 流程顺序

1. **合理性审查（先于改代码）**  
   - 判断是否合理、可落地。  
   - 有风险、歧义、漏洞、架构冲突 → **先澄清，不得直接开发**。  
   - 对齐后写入需求文档，再开发。

2. **按文档开发**  
   - 以定稿文档为准，避免口头漂移。

3. **交付前自检与修补**  
   - 后端：列表/详情、创建、更新、删除、校验、权限、软删、关联、错误返回。  
   - 前端：入口、表单/抽屉、只读边界、按钮显隐、提示、与 API 一致。  
   - 发现缺口当场补，再算完成。

### 1.2 约定沉淀与跨模块推广

- 需求适合固化为约定 → **与负责人确认后**写入约定文档（及 Agent rules）。  
- **双写**：项目需求文档中的可跨项目约定，须同步进本约定包（或项目内等价文件）；仅业务专用细节可只留需求文档并做映射链接。  
- 同一思路能提升其他模块 → **先提示范围与收益**，确认后再批量修补。

### 1.3 衍生需求提示

- 主需求本身 OK，但常连带相邻能力（如草稿/提交、查看/变更记录、字典中文）→ **主动提醒**。  
- 确认纳入后 **先写文档或待开发池**，再排期，减少返工。

### 1.4 待开发功能池

- 合理、能提高完成度，但暂无时间开发/测试 → 记入待开发池。  
- 不与「缺陷」「待确认」混放。

## 2. 数据库迁移双轨（强烈建议）

适用于「新开户要全量建表 + 老环境要幂等升级」的多租户 / 多环境系统。

| 场景 | 全量建表脚本（如 V1） | 幂等补列脚本（如 R__） |
|------|----------------------|------------------------|
| **新建表** | 完整 `CREATE TABLE`（含标准七列） | **不写**建表 |
| **已有表加列** | 同步补进 `CREATE TABLE` | **每列一行** `ADD COLUMN IF NOT EXISTS` |
| **索引** | 独立索引脚本（如 V2） | 一般不在补列脚本建索引 |
| **数据更正** | — | 独立可重复脚本（字典 / 回填 / 菜单） |
| **禁止** | — | 补列脚本里散落 `CREATE TABLE` / 一条 ALTER 多列；**禁止**每加一字段就新建一份版本化脚本 |

### 2.1 固定脚本槽位（强制习惯）

每个库/层只维护**固定少量文件**，内容增长写进槽位，不无限拆分：

| 职责 | 建议文件角色 |
|------|----------------|
| 建表 | V1（或等价全量建表） |
| 索引 | V2 |
| 一次性种子 / 历史注释 | V3 / V4（尽量冻结） |
| 补全字段 | R__（审计列可与业务列分文件，便于非事务逐条 ALTER） |
| 更正数据 | R__ data_fix（菜单、字典、回填） |

平台库 vs 业务库分目录；菜单幂等同步放 data_fix，勿再拆 `V5+`。

### 2.2 多 schema / search_path 防护

连接 `search_path = tenant, public` 时，租户**缺表**会导致 SQL 落到 public 同名表（静默串库）。约定：

- 启动/迁库后校验业务 schema 表齐全；缺表须阻断或强制补建。  
- 离线提供「对照全量建表脚本扫缺表」工具。  
- 勿向 public 写入租户业务 DDL。

### 2.3 提交前检查与锁表

- [ ] 新表字段在全量脚本齐全  
- [ ] 业务加列在补列脚本有对应 `IF NOT EXISTS`  
- [ ] 标准审计/软删列统一走审计补列槽，勿在业务补列脚本零散追加  
- [ ] 大批量 `ALTER` 勿塞进单事务长 DO；可 `executeInTransaction=false` 逐条提交

## 3. 标准审计七列与软删除（强制）

每张业务主表建议具备：

| 字段 | 类型建议 | 含义 |
|------|----------|------|
| `created_at` | 时间戳 | 创建时间 |
| `updated_at` | 时间戳 | 更新时间 |
| `created_by` | 用户 ID | 创建者 |
| `updated_by` | 用户 ID | 更新者 |
| `is_deleted` | 小整型，默认 0 | `0` 未删 / `1` 已删；存量未删行须回填为 `0`，列须 `DEFAULT 0`（尽量 `NOT NULL`） |
| `deleted_at` | 时间戳 | 删除时间 |
| `deleted_by` | 用户 ID | 删除者 |

**配套姓名快照（强制，§6.3）**：有对应 `*_by` 时须有 `created_by_name` / `updated_by_name` / `deleted_by_name`；写 UUID 的同一事务写展示名。

### 3.1 行为约定

| 操作 | 约定 |
|------|------|
| 删除 | **禁止**业务物理删主表；写软删三字段（含 `deleted_by_name`）；有 `is_active` 可同步停用 |
| 查询 | 默认过滤未删除；**列表/详情/下拉/统计/登录/存在性校验**一律排除已软删；联表时每张有软删列的表都要滤 |
| 唯一键撞软删行 | **恢复**该行（清空删除标记后 UPDATE），勿再 INSERT |
| 插入审计 | 填 `created_by` + `created_by_name`（及默认 `is_deleted=0`） |
| 更新审计 | `updated_at` / `updated_by` / `updated_by_name` **只由统一追加逻辑写一次** |
| UPDATE 禁写 | `id`、`created_*`、以及由框架托管的 `updated_*`；普通更新禁止客户端伪造软删字段与审计姓名 |
| softDelete 无列 | **禁止**静默降级为物理 DELETE；应报错并要求先补列 |

**读过滤例外**（须文档标明）：变更记录快照加载、按唯一键查找已软删行以便恢复、「含已删」管理排查接口。

### 3.2 明细表例外

「保存时先清空再写入」的子表明细行可物理删；**带业务唯一编码的主表必须软删**。平台「绑定关系先删后插」可视同明细例外。

### 3.3 防 UPDATE 列重复

客户端 body / 插入审计 **不要**预填 `updated_by` 再拼进 SET，又由框架追加第二次 → SQL 语法错误。统一：`strip` 客户端审计字段 + `appendUpdateAuditSets` 只写一次。

### 3.4 库结构 vs 行为门禁

| 目标 | 手段 |
|------|------|
| 表是否具备七列 / `is_deleted DEFAULT 0` / 存量回填 | **SQL 迁库**为真理来源 |
| 代码是否误物理删主表、读是否漏过滤 | **代码扫描 / Review**，不替代迁库 |

新建表必须在建表语句写出七列（含 `is_deleted NOT NULL DEFAULT 0`），禁止依赖运行时猜列补齐。不要用应用启动时 Java 扫库再 ALTER 作主方案；动态补列放在迁库 SQL 内。

## 4. 实体变更记录（主数据审计）

与「接口操作日志」分离：

| 类型 | 用途 |
|------|------|
| 操作日志 | 谁调了哪个 API |
| **实体变更记录** | 某条主数据字段怎么变 |

### 4.1 记录模型（推荐）

| 动作 | 内容 |
|------|------|
| create / update | 字段级 diff：`[{field, label, oldValue, newValue}]` |
| delete / submit / withdraw 等 | diff（如有）+ **精简快照** `snapshot_json` |
| 敏感字段 | 永不落库（password、token、*_hash 等） |

### 4.2 精简快照清单

- **一项主数据一张表**：字段名 / 类型 / 中文注释。  
- 扩展时：**先改文档清单，再改代码白名单**，二者必须一致。  
- 通用排除：`id`（已有 entity_id）、审计七列、大 JSON 权限树等。

### 4.3 新主数据强制

新增主数据表/类型时必须：

1. 纳入变更记录跟踪集合；  
2. 写路径（通用 CRUD / 专用 API）打点；  
3. 文档补充精简快照表。

## 5. 展示层约定

### 5.1 字典 / 状态码

- 库内存英文码；列表/详情用字典 **中文标签**。  
- 布尔 `is_*`：无字典时「是/否」；启用类字段可用「启用/停用」。  
- Schema 带 `dictType` 的字段必须走字典解析；页面进入时预加载相关字典。

### 5.2 外键显示

- 禁止列表/详情直接甩 UUID。  
- 配置：`linkTable` → 接口、名称键、编码兜底键。  
- `*_id` 外键的 `valueKey` 用实体 `id`；编码列另配独立键，勿混用。  
- 列表页挂载时预加载本页外键标签图。

### 5.3 API 与前端 ID

- JSON 中的 UUID **字符串化**输出，避免前端精度/类型问题。  
- 专用下拉/options 路径勿与 `/{id}` 通配冲突（例如用 `/options` 而非易冲突的 `/list`）。

### 5.4 列表勾选与批量作用域（强制）

凡列表（含弹窗多选）提供行勾选，须同时具备：

| 能力 | 要求 |
|------|------|
| 跨页缓存 | `row-key` + 跨页保留勾选；翻页/跳页不丢已选 |
| 全选当页 | 显式「全选当页」，叠加到跨页缓存（不清除其它页） |
| 取消全选 | 清空全部已选（含跨页） |
| 筛选重置 | 变更查询条件或重置时清空勾选，避免误操作 |

**导出 / 批量变更数据**（写库、生成简码、批量改状态等）：

1. 操作前弹出作用域选择：**仅勾选行** / **当前查询全部结果**。  
2. 未勾选时：默认仅允许「全部查询结果」，并二次确认；禁止静默对全量执行。  
3. 选「仅勾选行」时传已选 id 列表；选「全部」时传**当前筛选条件**（禁止只传当前页）。  
4. 后端按同一契约实现（`ids` vs `filters`/`keyword`/`all`）；全量须有合理上限或异步任务策略。  
5. **确认添加类**（从候选池勾选后写入）可只接受已选 id，不必提供「全部查询结果」。

无批量/导出能力的列表：**不要**挂空勾选列。

新列表默认按本条落地，不要等业务二次催。

## 6. 单据草稿 / 提交 / 撤回（业务模式）

适用于报修、申请类「提交后进流程」的单据（可按业务改名）：

| 概念 | 建议语义 |
|------|----------|
| **草稿** | 未提交；可改、可删；一般不占用下游资源 |
| **提交** | 进入流程状态；开始占用/通知等 |
| **撤回** | 仅「已提交且下游未处理」；回到草稿，可再改再提交 |
| **取消** | 流程作废；与撤回并存，勿混用文案 |

- 修改/提交/撤回/删除均写 **业务事件流水** + **实体变更记录**。  
- 禁止用通用 CRUD 绕过专用状态机接口。

### 6.1 主从单据保存

- 主从单据（计划/出入库/验收等）**禁止**「简单新增只存主表、明细丢失」。  
- 须专用保存（主表 + 明细同事务）或禁用会丢明细的简单新增入口。

### 6.2 业务子表关键字段冗余

- 设备相关业务的 **子表 / 事件流水 / 执行单**，在插入时从主单或设备台账拷贝：
  - **必有**：`device_id`（主单有设备时）
  - **建议**：`device_code`、`device_name`（写入时点快照，便于直查与导出）
- 子行 **不** 因台账事后改名自动级联；主单若改绑定设备，须同事务刷新或禁止改设备。
- 纯字典/模板配置表勿铺设备冗余。
- 按设备查询的表建 `device_id` 索引；老数据可回填。

### 6.3 工作人员姓名冗余（操作责任快照）**【强制】**

- **效力**：后续新功能 / 新表 / 新手写写点 **一律遵守**；存量缺口分批修补，禁止新增只写 UUID、不写姓名的入口。
- **目的**：操作记录背书、责任落实到人；避免用户改名后 join 导致追溯展示漂移。
- **取值**：优先真实姓名，空则回退登录名（如 `COALESCE(NULLIF(TRIM(real_name), ''), username)`）。
- **标准审计**：有 `created_by` / `updated_by` / `deleted_by` 时配套 `*_by_name`；写 UUID 的同一事务写姓名；软删恢复清空删除人姓名。
- **业务责任人**：报修人、派工/接单、确认人、经办人、审批人等有明确操作语义的字段，同步冗余 `{role}_name` / `{field}_name`。
- **快照语义**：写入时拷贝；**不**因事后改名级联更新历史行。
- **实现**：优先统一审计工具；手写 SQL 须显式解析并绑定姓名。客户端禁止伪造审计姓名列；老数据可按 UUID 回填。

## 7. 交付验收清单（通用）

### 7.1 后端改完

- [ ] 热加载/重启的是**改过的那个服务**  
- [ ] 经网关调列表/详情，字段名与前端 Schema 一致  
- [ ] 读接口已排除软删（含下拉/统计）；删走软删  

### 7.2 前端改完

- [ ] 硬刷新；Network 200；列表有数据  
- [ ] 列 `prop` 与 API 字段一致，无旧别名  
- [ ] 字典/外键显示中文，无裸 UUID  
- [ ] 有勾选列则具备跨页缓存、全选当页、取消全选；导出/批量变更先选作用域（§5.4）

### 7.3 库表改完

- [ ] 双轨脚本都改了（进固定槽位，未新建碎片脚本）  
- [ ] 迁库服务已跑；缺表扫描为 0  
- [ ] 相关业务服务已重启  

### 7.4 新增表 / 主数据（每次必做）

**库**

- [ ] 全量建表含标准七列 + 审计姓名三列（有 `*_by` 时）；索引进索引脚本；业务补列/种子进对应 R__  
- [ ] 业务责任人 UUID 字段同步建姓名快照列（§6.3）

**后端**

- [ ] 分页 / 保存 / 软删；可选 list/options、详情  
- [ ] 插入/更新/软删走统一审计（含 `*_by_name`）；手写 SQL 写用户 UUID 时同事务写姓名  
- [ ] 读接口排除软删；主数据须接入变更记录 + 快照白名单与文档一致  

**前端**

- [ ] Schema（list/required/dictType/linkTable）；外键 ref 配置；路由/菜单可访问  
- [ ] CRUD 与筛选走通；责任人展示优先读姓名快照列  
- [ ] 需要勾选时按 §5.4（跨页 / 全选当页 / 作用域）；无批量勿挂空勾选列

## 8. 非功能基线（可裁剪）

| 编号 | 类别 | 建议 |
|------|------|------|
| NF-01 | 性能 | 列表首屏有时限目标 |
| NF-02 | 性能 | 选择器分页/弹窗，避免全量下拉 |

| NF-04 | 可维护 | 新表优先通用 CRUD + Schema 配置；有状态机则禁用裸 CRUD |

| NF-06 | 编码 | 业务唯一编码创建后只读（前后端双重忽略改码） |
| NF-07 | 删除约束 | 存在关联业务数据则禁止删主数据（软删前校验） |

## 9. Agent / 人机协作规则（可做成 alwaysApply）

收到需求时默认：

1. 先审查合理性与衍生项；  
2. 确认后写文档再开发；  
3. 可沉淀约定 → 确认后写入本包（并双写项目需求附录）；  
4. 可跨模块推广 → 确认后批量修；  
5. 暂缓项 → `BACKLOG-*`；  
6. 交付按第 7 节自检（含 7.4）。
