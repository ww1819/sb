# 可复用工程约定包（从 MEIS 沉淀）

> **用途**：把 MEIS 实践中已验证、可迁移到其他系统的约定集中在此，减少重复劳动。  
> **来源**：`docs/meis-requirements.md` 附录 Q / C / D / E / F / G / H / I / R / S / T 等。  
> **用法**：新系统可整份复制后，按「落地映射」改路径与模块名；MEIS 专属细节见文末附录。

**版本**：1.23（2026-07-24）

---

## 维护约定（强制）

本文件与 `docs/meis-requirements.md` **双写**：

| 变更发生在 | 必须同步 |
|------------|----------|
| `meis-requirements.md` 新增/修订**可跨项目**约定 | 同步抽象进**本文**（去掉 MEIS 路径后写通用原则） |
| 本文修订可复用约定 | 若 MEIS 仍适用，回写需求文档对应附录，并更新 Q.6 索引 |

**可复用 vs MEIS 专用**：

- **写入本文**：协作流程、问题分流、库表双轨、审计七列/软删、实体变更记录模式、字典/外键展示、草稿/提交/撤回、验收清单、Agent 规则。
- **不必全文复制**：具体业务字段表（如 T.5）、某一工单状态机细项、某次审计补全记录 → 仅在文末「落地映射」链回需求文档。

## 0. 文档与问题分流（建议固定）

| 类型 | 放哪里 | 含义 |
|------|--------|------|
| 功能/业务需求 | 需求正文或附录 | 要做的能力 |
| 缺陷 / 技术债 | 「已知问题」章 | 已发现、要修 |
| 产品未决 | 「待确认问题」章 | 需业务方拍板 |
| 已认可但暂缓 | 「待开发功能池」章 | 值得做，暂无时间开发/测试 |
| 协作与技术约定 | 本约定包 / 项目附录 Q | 怎么协作、怎么写代码 |

**编号建议**：

- 需求：`{模块}-{类型}-{序号}`（F 功能 / B 规则 / X 问题…）
- 暂缓：`BACKLOG-{模块}-{序号}`

## 1. 需求协作与交付（强制）

### 1.1 流程顺序

1. **合理性审查（先于改代码）**  
   - 判断是否合理、可落地。  
   - 有风险、歧义、漏洞、架构冲突 → **先澄清，不得直接开发**。  
   - 对齐后写入需求文档，再开发。

2. **按文档开发**  
   - 以定稿文档为准，避免口头漂移。

3. **交付前自检与修补**  
   - 后端：列表/详情、创建、更新、删除、校验、权限、软删、关联、错误返回。  
   - 前端：入口、表单/抽屉、只读边界、按钮显隐、提示、与 API 一致。  
   - 发现缺口当场补，再算完成。

### 1.2 约定沉淀与跨模块推广

- 需求适合固化为约定 → **与负责人确认后**写入约定文档（及 Agent rules）。  
- **双写**：项目需求文档中的可跨项目约定，须同步进本约定包（或项目内等价文件）；仅业务专用细节可只留需求文档并做映射链接。  
- 同一思路能提升其他模块 → **先提示范围与收益**，确认后再批量修补。

### 1.3 衍生需求提示

- 主需求本身 OK，但常连带相邻能力（如草稿/提交、查看/变更记录、字典中文）→ **主动提醒**。  
- 确认纳入后 **先写文档或待开发池**，再排期，减少返工。

### 1.4 待开发功能池

- 合理、能提高完成度，但暂无时间开发/测试 → 记入待开发池。  
- 不与「缺陷」「待确认」混放。

## 2. 数据库迁移双轨（强烈建议）

适用于「新开户要全量建表 + 老环境要幂等升级」的多租户 / 多环境系统。

| 场景 | 全量建表脚本（如 V1） | 幂等补列脚本（如 R__） |
|------|----------------------|------------------------|
| **新建表** | 完整 `CREATE TABLE`（含标准七列） | **不写**建表 |
| **已有表加列** | 同步补进 `CREATE TABLE` | **每列一行** `ADD COLUMN IF NOT EXISTS` |
| **索引** | 独立索引脚本（如 V2） | 一般不在补列脚本建索引 |
| **数据更正** | — | 独立可重复脚本（字典 / 回填 / 菜单） |
| **禁止** | — | 补列脚本里散落 `CREATE TABLE` / 一条 ALTER 多列；**禁止**每加一字段就新建一份版本化脚本 |

### 2.1 固定脚本槽位（强制习惯）

每个库/层只维护**固定少量文件**，内容增长写进槽位，不无限拆分：

| 职责 | 建议文件角色 |
|------|----------------|
| 建表 | V1（或等价全量建表） |
| 索引 | V2 |
| 一次性种子 / 历史注释 | V3 / V4（尽量冻结） |
| 补全字段 | R__（审计列可与业务列分文件，便于非事务逐条 ALTER） |
| 更正数据 | R__ data_fix（字典、回填等非菜单） |
| 菜单目录 | **独立** R__ menus（唯一维护点；勿散落 data_fix / 临时脚本 / 业务库种子） |

平台库 vs 业务库分目录；菜单幂等同步放 **专用 menus 槽位**，勿再拆 `V5+`，也勿与 data_fix 混写。

### 2.2 多 schema / search_path 防护

连接 `search_path = tenant, public` 时，租户**缺表**会导致 SQL 落到 public 同名表（静默串库）。约定：

- 启动/迁库后校验业务 schema 表齐全；缺表须阻断或强制补建。  
- 离线提供「对照全量建表脚本扫缺表」工具。  
- 勿向 public 写入租户业务 DDL。

### 2.3 提交前检查与锁表

- [ ] 新表字段在全量脚本齐全  
- [ ] 业务加列在补列脚本有对应 `IF NOT EXISTS`  
- [ ] 标准审计/软删列统一走审计补列槽，勿在业务补列脚本零散追加  
- [ ] 大批量 `ALTER` 勿塞进单事务长 DO；可 `executeInTransaction=false` 逐条提交

## 3. 标准审计七列与软删除（强制）

每张业务主表建议具备：

| 字段 | 类型建议 | 含义 |
|------|----------|------|
| `created_at` | 时间戳 | 创建时间 |
| `updated_at` | 时间戳 | 更新时间 |
| `created_by` | 用户 ID | 创建者 |
| `updated_by` | 用户 ID | 更新者 |
| `is_deleted` | 小整型，默认 0 | `0` 未删 / `1` 已删；存量未删行须回填为 `0`，列须 `DEFAULT 0`（尽量 `NOT NULL`） |
| `deleted_at` | 时间戳 | 删除时间 |
| `deleted_by` | 用户 ID | 删除者 |

**配套姓名快照（强制，§6.3）**：有对应 `*_by` 时须有 `created_by_name` / `updated_by_name` / `deleted_by_name`；写 UUID 的同一事务写展示名。

### 3.1 行为约定

| 操作 | 约定 |
|------|------|
| 删除 | **禁止**业务物理删主表；写软删三字段（含 `deleted_by_name`）；有 `is_active` 可同步停用 |
| 查询 | 默认过滤未删除；**列表/详情/下拉/统计/登录/存在性校验**一律排除已软删；联表时每张有软删列的表都要滤 |
| 唯一键撞软删行 | **恢复**该行（清空删除标记后 UPDATE），勿再 INSERT |
| 主从明细软删再加 | 同主单下按业务键（如 `device_id`）撞已软删明细时：**恢复原行并 UPDATE**，写变更记录；勿新插重复键 |
| 插入审计 | 填 `created_by` + `created_by_name`（及默认 `is_deleted=0`） |
| 更新审计 | `updated_at` / `updated_by` / `updated_by_name` **只由统一追加逻辑写一次** |
| UPDATE 禁写 | `id`、`created_*`、以及由框架托管的 `updated_*`；普通更新禁止客户端伪造软删字段与审计姓名 |
| softDelete 无列 | **禁止**静默降级为物理 DELETE；应报错并要求先补列 |

**读过滤例外**（须文档标明）：变更记录快照加载、按唯一键查找已软删行以便恢复、「含已删」管理排查接口。

### 3.2 明细表例外

「保存时先清空再写入」的子表明细行可物理删；**带业务唯一编码的主表必须软删**。平台「绑定关系先删后插」可视同明细例外。

### 3.3 防 UPDATE 列重复

客户端 body / 插入审计 **不要**预填 `updated_by` 再拼进 SET，又由框架追加第二次 → SQL 语法错误。统一：`strip` 客户端审计字段 + `appendUpdateAuditSets` 只写一次。

### 3.4 库结构 vs 行为门禁

| 目标 | 手段 |
|------|------|
| 表是否具备七列 / `is_deleted DEFAULT 0` / 存量回填 | **SQL 迁库**为真理来源 |
| 代码是否误物理删主表、读是否漏过滤 | **代码扫描 / Review**，不替代迁库 |

新建表必须在建表语句写出七列（含 `is_deleted NOT NULL DEFAULT 0`），禁止依赖运行时猜列补齐。不要用应用启动时 Java 扫库再 ALTER 作主方案；动态补列放在迁库 SQL 内。

## 4. 实体变更记录（主数据审计）

与「接口操作日志」分离：

| 类型 | 用途 |
|------|------|
| 操作日志 | 谁调了哪个 API |
| **实体变更记录** | 某条主数据字段怎么变 |

### 4.1 记录模型（推荐）

| 动作 | 内容 |
|------|------|
| create / update | 字段级 diff：`[{field, label, oldValue, newValue}]` |
| delete / submit / withdraw 等 | diff（如有）+ **精简快照** `snapshot_json` |
| 敏感字段 | 永不落库（password、token、*_hash 等） |

### 4.2 精简快照清单

- **一项主数据一张表**：字段名 / 类型 / 中文注释。  
- 扩展时：**先改文档清单，再改代码白名单**，二者必须一致。  
- 通用排除：`id`（已有 entity_id）、审计七列、大 JSON 权限树等。

### 4.3 新主数据强制

新增主数据表/类型时必须：

1. 纳入变更记录跟踪集合；  
2. 写路径（通用 CRUD / 专用 API）打点；  
3. 文档补充精简快照表。

## 5. 展示层约定

### 5.1 字典 / 状态码

- 库内存英文码；列表/详情用字典 **中文标签**。  
- 布尔 `is_*`：无字典时「是/否」；启用类字段可用「启用/停用」。  
- Schema 带 `dictType` 的字段必须走字典解析；页面进入时预加载相关字典。

### 5.2 外键显示

- 禁止列表/详情直接甩 UUID。  
- 配置：`linkTable` → 接口、名称键、编码兜底键。  
- `*_id` 外键的 `valueKey` 用实体 `id`；编码列另配独立键，勿混用。  
- 列表页挂载时预加载本页外键标签图。

### 5.3 API 与前端 ID

- JSON 中的 UUID **字符串化**输出，避免前端精度/类型问题。  
- 专用下拉/options 路径勿与 `/{id}` 通配冲突（例如用 `/options` 而非易冲突的 `/list`）。

### 5.4 列表勾选与批量作用域（强制）

凡列表（含弹窗多选）提供行勾选，须同时具备：

| 能力 | 要求 |
|------|------|
| 跨页缓存 | `row-key` + 跨页保留勾选；翻页/跳页不丢已选 |
| 当页勾选 | 表头勾选即可选中当页；**不要**再单独做「已选 N 条 / 全选当页 / 取消全选」提示条 |
| 筛选重置 | 变更查询条件或重置时清空勾选，避免误操作 |

**导出 / 批量变更数据**（写库、生成简码、批量改状态等）：

1. 操作前弹出作用域选择：**仅勾选行** / **当前查询全部结果**。  
2. 未勾选时：默认仅允许「全部查询结果」，并二次确认；禁止静默对全量执行。  
3. 选「仅勾选行」时传已选 id 列表；选「全部」时传**当前筛选条件**（禁止只传当前页）。  
4. 后端按同一契约实现（`ids` vs `filters`/`keyword`/`all`）；全量须有合理上限或异步任务策略。  
5. **确认添加类**（从候选池勾选后写入）可只接受已选 id，不必提供「全部查询结果」。

无批量/导出能力的列表：**不要**挂空勾选列。

新列表默认按本条落地，不要等业务二次催。

### 5.5 列表筛选：多选与分类模糊拆分

高频维度（分类、科室、状态等）的列表筛选建议：

| 模式 | 约定 |
|------|------|
| **多选** | 前端多选下拉；请求参数为 **逗号分隔** UUID/码；后端 `IN (...)`；单值亦走同一路径 |
| **分类拆分** | 使用频次高的分类（设备/资产/财务等）拆成两条件：**多选 ID** + **编码/名称（可选首拼）模糊**；二者可同时 AND |
| **勿混用** | 模糊框不要兼做「选一条再查 ID」；多选负责精确集合，模糊负责宽检索 |
| **组件** | 字典/外键筛选支持 `multiple`；更多检索外键同理；提交前数组 `join(',')` |

新列表优先按本条设计；存量页分批推广。

### 5.6 外键选择回填（模板→单据等）

当表单外键选定后需带出关联实体若干字段（如选模板带周期/类型）：

| 项 | 要求 |
|----|------|
| 配置 | Schema 字段声明 `fillFromLink`：`{ 表单字段: 关联实体字段 }` |
| 数据 | 变更外键时按 `refSelect` 详情 URL `GET …/{id}` 取完整行再映射 |
| 时机 | **仅用户变更**外键时回填；打开编辑的首次赋值不覆盖 |
| 可改 | 回填为默认，允许再改；再次更换外键则覆盖映射字段 |
| 清空 | 清空外键时**不**强制清空已回填字段（防误触） |

派生只读字段（如周期天数由类型×值算出）：表单展示但禁止手改；变更依赖字段时前端即时同步；**保存时后端按同一规则重算落库**。

### 5.7 列表操作功能分列

操作入口超过约 3 个（审核、明细、编辑、删除等）时：

| 项 | 要求 |
|----|------|
| 分列 | 每个功能独立列（右固定），**不要**挤进单一「操作」列 |
| 显隐 | 按单据状态控制（如已审核隐藏编辑/删除）；无权限时显示「—」而非空白乱跳 |
| 实现 | 隐藏默认操作列 + `#extra-columns`（或等价）自挂列 |

周期类明细「下次到期」：添加行或改「上次完成」时前端即时算；保存时后端若为空按同一公式补齐（基准日=上次完成‖今天 + 周期天数）。

### 5.8 已审核头表 vs 明细

审核通过后：

| 能力 | 约定 |
|------|------|
| 头表 | **禁止**编辑业务字段、**禁止**删除整单（前后端） |
| 明细 | 若业务允许调整范围：单独入口（如「设备明细」）可增删改；保存只更新明细 |
| 软删明细再加 | 见 §3「主从明细软删再加」 |

### 5.9 计划/执行类单据头表紧凑排布

主从单据详情若头表过高导致明细需滚动才可见：

| 项 | 要求 |
|----|------|
| 标签 | **左置**（勿默认置顶两列撑高） |
| 密度 | 表单 `size=small`；收紧分组间距 |
| 排布 | 优先 **流式**（`flex-wrap`）：控件按内容宽度排列换行，输入框缩窄（约 128–168px），勿用等分栅格把控件拉满行宽 |
| 分组 | 建议：基本信息 / 执行与途径 / 制单审核；字段须声明 `group`，避免整块落入「其他」 |
| 只读空值 | 字典/途径只读时空值显示「—」，**勿**用空下拉占位「请选择」 |
| 列表列 | 计划/执行列表补齐业务识别列（单号、模板、状态、责任人、制单人时等）；列表用中文名列，表单用外键选择 |

### 5.10 多端途径（channel）字段展示

单据若写入 `*_channel`（取值如 `web` / `app` / `mp`）：

| 项 | 要求 |
|----|------|
| **必显** | 列表与详情须展示对应中文标签（字典或固定映射）；空值显示「—」 |
| **分层** | 头表途径（制单/提交/审核/删除等）与明细途径（执行/确认等）分开展示，勿混在一列 |
| **场景** | 详情「执行/办理」态若隐藏头表表单，仍须用摘要行常显头表途径 |
| **范围** | 有字段则显；无字段的业务不强加 |

落地映射：MEIS OPS.16.10 / 16.26 / 16.27 / 16.28（含纳入申请申请途径/确认途径）。

### 5.11 本机外设（高拍仪等）经本地代理

USB 外设（如文档高拍仪）**勿**由浏览器直连硬件；经厂商本机代理服务（常见 WebSocket `127.0.0.1:端口`）：

| 项 | 要求 |
|----|------|
| **封装** | 前端薄封装（连接/预览/拍照/关闭）；勿整页拷贝厂商 demo |
| **上传** | 外设产物（base64/文件）走系统统一文件上传 API，业务只存 URL |
| **降级** | 无本机服务时友好提示，并保留普通上传/手机相机入口 |
| **范围** | 仅工作站浏览器；移动端不接本机代理 |

落地映射：MEIS PLT-CAM-01（新良田 Eloam `ws://127.0.0.1:9000`）。开发机注意：Eloam 与对象存储（MinIO）**勿共占 9000**；MEIS 本地 MinIO 默认 API **9100**。

## 6. 单据草稿 / 提交 / 撤回（业务模式）

适用于报修、申请类「提交后进流程」的单据（可按业务改名）：

| 概念 | 建议语义 |
|------|----------|
| **草稿** | 未提交；可改、可删；一般不占用下游资源 |
| **提交** | 进入流程状态；开始占用/通知等 |
| **撤回** | 仅「已提交且下游未处理」；回到草稿，可再改再提交 |
| **取消** | 流程作废；与撤回并存，勿混用文案 |

- 修改/提交/撤回/删除均写 **业务事件流水** + **实体变更记录**。  
- 禁止用通用 CRUD 绕过专用状态机接口。

### 6.1 主从单据保存

- 主从单据（计划/出入库/验收等）**禁止**「简单新增只存主表、明细丢失」。  
- 须专用保存（主表 + 明细同事务）或禁用会丢明细的简单新增入口。

### 6.4 草稿保存后询问是否提交

- 适用于「先落草稿、再决定是否进流程」的申请类单据（Web / App 一致）。
- 用户点确定 → **先持久化为草稿** → 弹窗「是否提交」：是则走提交接口，否则留草稿。
- 草稿列表须支持编辑 / 删除 / 提交；已提交且未进入下游处理的须支持 **撤回** 回草稿（语义见 §6）。

### 6.5 业务附件（图片）上限

- 业务单据附件字段建议存 **URL 数组**（或等价 JSON），经统一文件上传接口落对象存储。
- 须在需求中明确：是否必传、张数/大小上限；后端保存时校验上限，前端同步限制。
- 草稿可改附件；进入流程后只读（除非业务另定）。

### 6.2 业务子表关键字段冗余

适用于设备相关子表，以及**一切主从业务单据**的明细/更深子表。

**A. 关联主数据（有 `*_id` 时）**

- 插入时从主单或主数据拷贝：
  - **必有**：外键 id（如 `device_id`）
  - **必有（有业务展示价值时）**：对应 **编码和/或名称**（如 `device_code`/`device_name`；科室/仓库/供应商/模板等同理）
- 子行 **不** 因主数据事后改名自动级联；主单若改绑定对象，须同事务刷新或禁止改绑。
- 纯字典/模板配置关系表勿铺展示冗余。
- 高频按对象查询的表建对应 `*_id` 索引；老数据可回填。

**B. 单据头下沉（新单据强制）**

- **强制**：明细及向下子表冗余主单 **业务单号**（`plan_no` / `execution_no` / `entry_no` / `wo_no` 等），便于直查与导出。
- **建议**：按查询需要冗余少量头字段快照（模板名称、计划日期、级别/类别、责任科室等）；**禁止**整表头拷贝。
- **默认不冗余**可变状态机字段到明细（除非产品明确要历史快照语义）。
- 写入与主单保存同事务；与单号生成约定（§6.8）配合：主单取号后写入明细。

### 6.3 工作人员姓名冗余（操作责任快照）**【强制】**

- **效力**：后续新功能 / 新表 / 新手写写点 **一律遵守**；存量缺口分批修补，禁止新增只写 UUID、不写姓名的入口。
- **目的**：操作记录背书、责任落实到人；避免用户改名后 join 导致追溯展示漂移。
- **取值**：优先真实姓名，空则回退登录名（如 `COALESCE(NULLIF(TRIM(real_name), ''), username)`）。
- **标准审计**：有 `created_by` / `updated_by` / `deleted_by` 时配套 `*_by_name`；写 UUID 的同一事务写姓名；软删恢复清空删除人姓名。
- **业务责任人**：报修人、派工/接单、确认人、经办人、审批人等有明确操作语义的字段，同步冗余 `{role}_name` / `{field}_name`。
- **快照语义**：写入时拷贝；**不**因事后改名级联更新历史行。
- **实现**：优先统一审计工具；手写 SQL 须显式解析并绑定姓名。客户端禁止伪造审计姓名列；老数据可按 UUID 回填。

### 6.6 统一业务打印流水（标签/单据）

- **一张流水表**承载多来源打印（台账、盘点补打、日后出入库等），避免每业务一张打印表。
- **必有**：实体主键、关键快照（如设备编码/名称）、打印人 UUID + **姓名快照**（§6.3）、打印时间、模板编码。
- **业务上下文（可空，兼容纯主数据打印）**：`biz_type`、`biz_id`、`biz_no`、`biz_item_id`。
- 单据侧若维护「需补打 / 已打印 / 次数」，打印成功须同事务回写，并清「需补打」。
- 主从保存若重写明细，须 **保留明细 id**（upsert），否则流水 `biz_item_id` 与次数会失联。

### 6.7 多端并发单据：事件流水 + 明细字段变更 + 乐观锁

适用于 App/Web（或多端）同时填写同一执行单/作业单明细的场景。

- **与接口审计、主数据变更记录分离**：`operation_log` 记 API；主数据 `entity_change_log` 记主档；单据执行另建 **业务事件流水** + **明细/内容字段变更**。
- **业务事件**：生成、提交、撤回、审核、作废、认领等（操作人 UUID+姓名、端、单据 id/no、时间）。
- **字段变更**：针对明细行/内容行，记 `entity_type`、`entity_id`、`field`、old/new、操作人+姓名、端、时间。
- **乐观锁**：明细与内容行带 `row_version`（或等价）；冲突时明确失败，禁止静默覆盖。
- **状态锁**：提交/审核后明细只读（除非业务允许撤回后再编）。
- 多业务同构时可共享一张单据变更表并用 `module` 区分；**业务执行表是否物理分表由产品决定**，与本追溯约定正交。

### 6.8 业务单号系统生成、禁止手改

适用于计划单、执行单、入库单等业务单据编号：

- **新建**：后端按「前缀 + yyyyMMdd + 日流水」自动取号；**忽略**客户端传入的单号。
- **更新**：单号字段只读；后端 UPDATE **不得**用请求体覆盖已有单号。
- **前端**：表单 `readonly`；列表可展示。
- **通用 CRUD**：若表暴露通用接口，创建时强制取号，更新时剥离单号字段，防止绕过专用保存。
- 具体前缀按业务模块在需求中定稿（勿多模块共用同一前缀+同表列以外的冲突规则）。

- [ ] 双轨脚本都改了（进固定槽位，未新建碎片脚本）

**库**

## 9. Agent / 人机协作规则（可做成 alwaysApply）

收到需求时默认：

1. 先审查合理性与衍生项；  
2. 确认后写文档再开发；

5. 暂缓项 → `BACKLOG-*`；  
6. 交付按第 7 节自检（含 7.4）。
